# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

macOS app that captures system audio output via **Core Audio Taps** (macOS 14.2+ API, no virtual audio device required) and renders Windows XP Media Player–style visualizations in Metal. Three scenes: Bars, Scope, Alchemy (compute-shader particles). English + Spanish localization with live runtime switching.

## Build / test / run commands

The repo is a hybrid: a SwiftPM package (`Domain` + `Application` Swift libraries plus `TPCircularBuffer` C target) and an Xcode app target generated by [XcodeGen](https://github.com/yonaskolb/XcodeGen) from `project.yml`.

```bash
# Domain + Application tests (pure Swift, no Apple frameworks beyond Foundation, <1s)
swift test
swift test --filter DomainTests.LanguageTests        # single test class
swift test --filter ApplicationTests                  # whole target

# Whole app (requires XcodeGen + macOS 14.2 SDK)
xcodegen generate                                     # regenerate .xcodeproj from project.yml
xcodebuild -project AudioVisualizer.xcodeproj -scheme AudioVisualizer -destination 'platform=macOS' build
xcodebuild test -project AudioVisualizer.xcodeproj -scheme AudioVisualizer -destination 'platform=macOS'

# Run a single Xcode test:
xcodebuild test -project AudioVisualizer.xcodeproj -scheme AudioVisualizer -destination 'platform=macOS' \
  -only-testing:AudioVisualizerTests/BundleLocalizerTests/test_spanish_returns_spanish_strings

# Launch a freshly-built debug build
open ~/Library/Developer/Xcode/DerivedData/AudioVisualizer-*/Build/Products/Debug/AudioVisualizer.app
```

**Regenerate the Xcode project (`xcodegen generate`) any time you add or move a source file under `AudioVisualizer/`, change `project.yml`, or modify `Package.swift`.** XcodeGen builds the entire `.xcodeproj` from scratch — your changes to source files don't require a regen, but adding a new file does.

## Streaming live diagnostic logs

The app emits structured `os.log` under subsystem `dev.audiovideogen.AudioVisualizer` with categories `capture`, `analysis`, `render`, `vm`. Stream them while the app is running:

```bash
# Note: zsh has a `log` builtin. Use the absolute path.
/usr/bin/log stream --predicate 'subsystem == "dev.audiovideogen.AudioVisualizer"' --info --style compact

# One-shot, last 5 minutes
/usr/bin/log show --last 5m --predicate 'subsystem == "dev.audiovideogen.AudioVisualizer"' --info --debug
```

When debugging a "Waiting for audio…" problem, look at the `capture` category's per-second `io: callbacks/s=N frames/s=N peakAmp=…` lines. They tell you whether the IOProc is firing (`callbacks/s > 0`) and whether non-silent samples are reaching the ring buffer (`peakAmp > 0`).

## Architecture (Clean + DDD)

```
Sources/Domain/        — pure Swift, only `import Foundation` allowed
Sources/Application/   — use cases, depends only on Domain
AudioVisualizer/
  Infrastructure/      — Apple framework adapters (CoreAudio, Analysis, Metal, Persistence, Localization, Logging)
  Presentation/        — SwiftUI views + @Observable view models
  App/                 — @main entry point + CompositionRoot (the single wiring point)
Vendor/TPCircularBuffer/ — BSD lock-free ring buffer (C), exposed as SwiftPM target
docs/superpowers/      — design specs + implementation plans for each release
```

**Architectural invariant** (enforced by reading code, not by tooling):

```bash
grep -rE "import (CoreAudio|AVFoundation|Metal|MetalKit|Accelerate|SwiftUI|AppKit)" Sources/Domain Sources/Application
```

…must return zero matches. Domain and Application stay pure Swift so the test suite for them runs in <1 s without macOS frameworks. **If you find yourself wanting to import an Apple framework in `Sources/Domain` or `Sources/Application`, you are crossing a boundary — refactor instead.**

### Bounded contexts and their ports

Each Domain context defines protocols (ports) that Infrastructure adapters implement. The Composition Root (`AudioVisualizer/App/CompositionRoot.swift`) is the only place that constructs concrete adapters and hands them to use cases. **Adding a new feature touches every layer in the same direction**: Domain types → Application use case → Infrastructure adapter → Presentation view → CompositionRoot wiring.

| Context | Port (Domain) | Adapter (Infrastructure) |
|---|---|---|
| AudioCapture | `SystemAudioCapturing` | `CoreAudio/CoreAudioTapCapture` |
| AudioCapture | `ProcessDiscovering` | `CoreAudio/RunningApplicationsDiscovery` |
| AudioCapture | `PermissionRequesting` | `CoreAudio/TCCAudioCapturePermission` |
| AudioAnalysis | `AudioSpectrumAnalyzing` | `Analysis/VDSPSpectrumAnalyzer` |
| AudioAnalysis | `BeatDetecting` | `Analysis/EnergyBeatDetector` |
| Visualization | `VisualizationRendering` | `Metal/MetalVisualizationRenderer` |
| Preferences | `PreferencesStoring` | `Persistence/UserDefaultsPreferences` |
| Localization | `Localizing` | `Localization/BundleLocalizer` (`@Observable`) |

### Critical non-obvious wiring rules

**The Core Audio IOProc must dispatch on `nil` (Core Audio's IO thread), not a custom queue.** Earlier versions passed `drainQueue` to `AudioDeviceCreateIOProcIDWithBlock`, but `drainQueue` was already being held by the drainer's infinite `while` loop — the IOProc never fired and the visualizer was permanently silent. See [CoreAudioTapCapture.swift](AudioVisualizer/Infrastructure/CoreAudio/CoreAudioTapCapture.swift) `AudioDeviceCreateIOProcIDWithBlock(&newProc, aggID, nil) { … }`. The IOProc body is RT-safe (pointer arithmetic + `TPCircularBuffer.write` + atomic counter writes only — no Swift runtime calls, no allocations, no locks).

**System-wide capture uses `CATapDescription(stereoGlobalTapButExcludeProcesses: [])`, not `stereoMixdownOfProcesses: []`.** The latter taps zero processes; the former taps everything on the default output.

**Per-app capture resolves the audio process by *parent bundle ID* at `start()` time, not by stored PID.** Chrome's audio helper is ephemeral — it restarts under new PIDs. `RunningApplicationsDiscovery.parentBundleID(of:)` strips `.helper.*` suffixes so `com.google.Chrome.helper.Audio` and `com.google.Chrome.helper.Plugin` both match the parent `com.google.Chrome`. Multiple matches → multi-process tap. Zero matches → silent fall-back to global tap.

**The CompositionRoot deliberately constructs `MetalVisualizationRenderer.make()` (static factory), not `init()`.** `NSObject.init()` cannot be overridden as `throws`, so the renderer exposes a throwing factory instead.

## Adding a new user-facing string

1. Add a case in `Sources/Domain/Localization/ValueObjects/L10nKey.swift` with a stable `rawValue` (use the dotted-namespace convention, e.g. `"toolbar.foo.label"`).
2. Add translations to `AudioVisualizer/Resources/Localizable.xcstrings` for **every** locale (`en` and `es` today). Missing keys fall back to the raw key string in the UI — they don't crash, but they look broken.
3. Read it in views via `localizer.string(.yourNewKey)`. Views must declare `@Bindable var localizer: BundleLocalizer` or `let localizer: Localizing` so `@Observable` invalidation propagates on language switch.

## Plan / spec convention

Every meaningful change is preceded by a design spec in `docs/superpowers/specs/YYYY-MM-DD-<feature>-design.md` and an implementation plan in `docs/superpowers/plans/YYYY-MM-DD-<feature>.md`. The plans use TDD-shaped task lists with exact code and expected build/test output for each step. Subsequent agents executing tasks should reference these — they encode decisions that aren't in the source (e.g. why we use bundle-ID matching instead of PID).

## Project constraints

- Minimum macOS: **14.2** (`CATapDescription` API), declared in both `Package.swift` and `project.yml`.
- The app is sandboxed (`com.apple.security.app-sandbox`) with `com.apple.security.device.audio-input` and triggers a one-time TCC "Audio Capture" prompt the first time a tap is created.
- No virtual audio device (no BlackHole, no kext) — that's a deliberate user-experience choice baked into the architecture.
- No microphone use ever.
